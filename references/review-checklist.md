# Review Checklist — 畫圖前自我審查 + 現有圖審查

## 畫圖前：一致性審查（Phase 4 執行）

確認視角後，生成 SVG 前必須過一遍：

```
□ 所有節點是否符合本視角的元素白名單？
  → 系統架構圖：只有系統/DB/外部介面，沒有步驟框或角色
  → 資料流圖：只有資料實體與讀寫箭頭，沒有系統部署節點
  → 作業流圖：只有角色/步驟/決策，沒有系統節點或 DB

□ 所有箭頭是否符合本視角的語意？
  → 系統架構圖：箭頭 = 系統呼叫，不是步驟順序
  → 資料流圖：箭頭 = 資料流向，不是系統依賴
  → 作業流圖：箭頭 = 步驟順序，不是系統呼叫

□ 並列且無呼叫關係的節點，是否沒有箭頭連接？
  → 「同屬一個 Squad」不等於「有箭頭」
  → 「流程上相鄰」不等於「有系統依賴」

□ 待確認的介面是否用虛線標示？
  → 方向或強度未確認 → 虛線箭頭 + [待確認]
  → 已確認 → 實線箭頭

□ 跨 Squad 介面的方向（provides_to / depends_on）是否已確認？
  → 若未確認，標 [待確認]，不推測方向
```

任何一項未通過 → 修正後再輸出，不帶疑問往下畫。

## 現有圖審查：視角混用偵測（Phase 5 執行）

當使用者貼上現有圖要求審查時，依序執行：

### Step 1：判斷主要視角

從圖的元素語言推斷使用者原本想畫哪種圖：

| 若圖中有這些元素 | 推斷主要視角 |
|----------------|------------|
| 系統節點 + 部署邊界框 + API 標示 | 系統架構圖 |
| 資料實體 + 讀寫標注 + ownership | 資料流圖 |
| 角色 swimlane + 步驟框 + 決策菱形 | 作業流圖 |
| 上述混合出現 | 混合視角（需拆分） |

### Step 2：列出混用位置

```
## 圖的視角審查

判斷現有圖的主要視角：[系統架構圖 / 資料流圖 / 作業流圖 / 混合]

發現以下視角混用問題：

| 位置 | 混用的元素 | 屬於哪種視角 | 修正建議 |
|------|-----------|------------|---------|
| [節點/箭頭描述] | [元素名稱] | [視角] | 刪除 / 改為 / 移到另一張圖 |
```

### Step 3：常見混用模式

| 混用模式 | 症狀 | 修正方向 |
|---------|------|---------|
| 架構圖 + 作業流 | 系統節點之間用「建檔→估算→上架」順序連接 | 架構圖只保留系統節點，拆出作業流圖 |
| 架構圖 + 資料流 | 系統節點上標了讀寫權限或 ownership | 架構圖標系統名稱，DFD 另畫資料流向 |
| 作業流 + 系統節點 | swimlane 裡畫了 DB 節點或 API 標示 | 步驟框備注「在哪個系統操作」，不畫系統節點 |
| 箭頭語意混亂 | 同一張圖的箭頭有時代表順序、有時代表呼叫 | 確認主要視角後統一箭頭語意 |

### Step 4：輸出格式

```
## 圖的視角審查結果

主要視角（推斷）：系統架構圖
混用問題：[N] 處

[問題 1]
位置：「建檔 → 估算 → 上架」箭頭序列
混用元素：作業流的步驟順序箭頭
修正建議：系統架構圖不表達作業順序，這三個步驟若屬於同一個後台系統，
          在架構圖中合併為一個系統節點；若要保留步驟細節，另畫作業流圖。

[問題 2]
位置：證照系統 → 領隊查詢 箭頭
混用元素：無呼叫關係的兩個模組之間不應有箭頭
修正建議：刪除此箭頭，兩者並列即可。

---
確認修正方向後，我會重畫。
```
