# 資料流圖（Data Flow Diagram, DFD）

載入時機：Phase 1 確認視角為「資料流圖」後讀取。

---

## 定義

描述資料的流動與擁有關係：資料從哪裡產生、流向哪裡、誰擁有、誰有讀寫權限。

回答的核心問題：**這筆資料從哪產生？誰 own 它？流向哪些系統？誰讀、誰寫？**

---

## 元素白名單

```
✅ 允許
- 資料實體節點（標資料名稱 + owner Squad）
- 資料流向箭頭（標 R / W / RW / Event）
- DB / Table 節點（標 DB 名稱與 ownership）
- 資料產生點（Write source）與消費點（Read consumer）
- 資料更新的觸發條件（事件觸發 / 定時觸發）
- 資料轉換說明（ETL、mapping 邏輯）

❌ 禁止
- 系統部署節點（那是架構圖的元素）
- API 呼叫細節（endpoint、協定版本號）
- 人的操作行為
- 作業步驟順序
- 決策菱形
```

---

## 箭頭語意

- 箭頭方向：**資料從 A 流向 B**
- 必須標注讀寫方向（R = 讀取、W = 寫入、RW = 讀寫）
- 雙向箭頭需標明哪個方向是主動觸發

---

## Ownership 標示規則

- 每個資料實體必須標明 owner Squad
- 多個 Squad 可讀但只有一個可寫：標 `W: S4 / R: S5, S6`
- 雙方都可寫時需確認主從關係，標 `Primary W: S4 / Secondary W: S6`
- Ownership 未確認：標 `[ownership 待確認]`，不推測

---

## 常見錯誤

- 把系統節點（SERP 後台、LionGroupRPM）當資料實體 → 資料實體是「資料」，不是「系統」
- 箭頭沒有標 R/W → 不清楚誰在讀誰在寫，是 DFD 最常見的遺漏
- Ownership 空白 → 每個節點都必須有 owner，不確定就標 [待確認]
- 把 API 呼叫細節畫進來 → API 是系統架構圖的元素
