# 資料流圖（Data Flow Diagram, DFD）

載入時機：Phase 1 確認視角為「資料流圖」後讀取。

---

## 定義

描述資料的流動與擁有關係：資料從哪裡產生、流向哪裡、誰擁有、誰有讀寫權限。

回答的核心問題：**這筆資料從哪產生？誰 own 它？流向哪些系統？誰讀、誰寫？**

---

## 元素白名單

```
✅ 允許
- 資料實體節點（標資料名稱 + owner Squad）
- 資料流向箭頭（標 R / W / RW / Event）
- DB / Table 節點（標 DB 名稱與 ownership）
- 資料產生點（Write source）與消費點（Read consumer）
- 資料更新的觸發條件（事件觸發 / 定時觸發）
- 資料轉換說明（ETL、mapping 邏輯）

❌ 禁止
- 系統部署節點（那是架構圖的元素）
- API 呼叫細節（endpoint、協定版本號）
- 人的操作行為
- 作業步驟順序
- 決策菱形
```

---

## 箭頭語意

- 箭頭方向：**資料從 A 流向 B**
- 必須標注讀寫方向（R = 讀取、W = 寫入、RW = 讀寫）
- 雙向箭頭需標明哪個方向是主動觸發

---

## Ownership 標示規則

- 每個資料實體必須標明 owner Squad
- 多個 Squad 可讀但只有一個可寫：標 `W: S4 / R: S5, S6`
- 雙方都可寫時需確認主從關係，標 `Primary W: S4 / Secondary W: S6`
- Ownership 未確認：標 `[ownership 待確認]`，不推測

---

## 佈局建議（左→右資料流向）

- **左側**：資料來源（外部輸入、Write source、上游 Squad 提供的原料）
- **中央**：核心資料實體（本 Squad own 的主要資料）
- **右側**：資料消費者（Read consumer、下游 Squad 接收的輸出）
- **回寫路徑**：用不同色或虛線區分，避免與正向流混用，並標明觸發條件
- 同一 Squad own 的資料實體放在同一個虛線區塊內，跨 Squad 的資料實體放在區塊外

---

## 常見錯誤（含輸入信號辨識）

每條格式：**[使用者說了什麼]** → [錯誤傾向] → [正確做法]

- **使用者說「把 SERP 後台、LionGroupRPM 這些系統畫出來」**
  → 容易把系統節點當資料實體放進 DFD
  → DFD 的節點是「資料」不是「系統」；系統是資料的載體，不是主角

- **使用者沒說讀還是寫，只說「A 跟 B 有關係」**
  → 容易畫出沒有 R/W 標注的箭頭
  → DFD 每條箭頭必須標 R / W / RW / Event；不確定就問清楚再畫

- **使用者說「這個資料兩邊都會用到」但沒說誰 own**
  → 容易空白 ownership 或隨意推測
  → 每個節點必須標 owner，不確定標 `[ownership 待確認]`，不推測

- **使用者說「串接的 API 細節也一起畫進來」**
  → 容易把 endpoint、協定版本號畫進 DFD
  → API 呼叫細節屬於系統架構圖；DFD 只關心資料本身，不關心傳輸協定
