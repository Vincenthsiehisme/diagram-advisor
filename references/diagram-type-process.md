# 作業流圖（Operation / Process Flow）

載入時機：Phase 1 確認視角為「作業流圖」後讀取。

---

## 定義

描述人的操作行為：誰在什麼時間點、做什麼動作、遇到什麼判斷條件、交給誰繼續。

回答的核心問題：**誰在什麼步驟做什麼操作？判斷點在哪？角色之間怎麼交接？**

---

## 元素白名單

```
✅ 允許
- 角色 swimlane（業務、OP、PM、客戶、領隊）
- 操作步驟框（動詞 + 受詞，主詞是人）
- 決策菱形（人的判斷點，是 / 否分支）
- 觸發條件（什麼事件啟動流程）
- 開始 / 結束節點
- 步驟間的順序箭頭（時間順序）
- 括號備注（「在 SERP 後台操作」），說明操作入口

❌ 禁止
- 系統節點（LionGroupRPM、SERP 後台）
- DB 節點
- API 介面標示
- Squad 邊界框
- 資料 ownership 說明
```

---

## 箭頭語意

- 箭頭方向：**步驟的時間順序**，「先做 A 再做 B」
- 不代表系統呼叫，不代表資料流向
- 跨 swimlane 的箭頭代表任務交接給另一個角色

---

## 步驟框寫法規則

- 主詞是「人」，不是「系統」
  - ✅ 「OP 審核配額」
  - ❌ 「系統扣除配額」（系統自動行為不畫步驟框）
- 若需標明操作入口，用括號備注：「OP 審核配額（SERP 後台）」
- 系統自動觸發的行為：用觸發條件節點表示，不畫決策菱形
- 只有**人的判斷**才畫決策菱形；系統自動判斷不畫

---

## 常見錯誤

- 把「系統自動執行」的步驟畫成操作步驟框 → 作業流只畫人的操作
- 把系統節點（DB、API）畫進來 → 作業流不關心系統內部結構
- 決策菱形代表系統判斷 → 只畫人的判斷點
- 跨角色交接沒有明確的 swimlane 分界 → 每個角色一條 lane，交接點跨線
