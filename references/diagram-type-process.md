# 作業流圖（Operation / Process Flow）

載入時機：Phase 1 確認視角為「作業流圖」後讀取。

---

## 定義

描述人的操作行為：誰在什麼時間點、做什麼動作、遇到什麼判斷條件、交給誰繼續。

回答的核心問題：**誰在什麼步驟做什麼操作？判斷點在哪？角色之間怎麼交接？**

---

## 元素白名單

```
✅ 允許
- 角色 swimlane（業務、OP、PM、客戶、領隊）
- 操作步驟框（動詞 + 受詞，主詞是人）
- 決策菱形（人的判斷點，是 / 否分支）
- 觸發條件（什麼事件啟動流程）
- 開始 / 結束節點
- 步驟間的順序箭頭（時間順序）
- 括號備注（「在 SERP 後台操作」），說明操作入口

❌ 禁止
- 系統節點（LionGroupRPM、SERP 後台）
- DB 節點
- API 介面標示
- Squad 邊界框
- 資料 ownership 說明
```

---

## 箭頭語意

- 箭頭方向：**步驟的時間順序**，「先做 A 再做 B」
- 不代表系統呼叫，不代表資料流向
- 跨 swimlane 的箭頭代表任務交接給另一個角色

---

## 步驟框寫法規則

- 主詞是「人」，不是「系統」
  - ✅ 「OP 審核配額」
  - ❌ 「系統扣除配額」（系統自動行為不畫步驟框）
- 若需標明操作入口，用括號備注：「OP 審核配額（SERP 後台）」
- 系統自動觸發：用**虛線箭頭**連接，箭頭上標注「系統觸發」或「自動」，不畫步驟框，不畫決策菱形
  - 範例：「確認名單」步驟框 --虛線「系統觸發」--> 「拋轉至 S6」標注框（非步驟框）
- 只有**人的判斷**才畫決策菱形；系統自動判斷不畫

---

## 常見錯誤（含輸入信號辨識）

每條格式：**[使用者說了什麼]** → [錯誤傾向] → [正確做法]

- **使用者說「系統自動扣除配額」「系統判斷庫存是否足夠」**
  → 容易把系統自動行為畫成步驟框或決策菱形
  → 作業流只畫人的操作；系統自動行為用虛線箭頭 + 「系統觸發」標注，不畫步驟框

- **使用者說「也把 DB 跟 API 畫進來讓 RD 看清楚」**
  → 容易在 swimlane 裡混入系統節點或 DB
  → 作業流不關心系統內部結構；若需標明操作入口，用括號備注「（在 SERP 後台操作）」即可

- **使用者說「這個判斷是系統自動跑的規則」但要求畫決策菱形**
  → 容易把系統自動判斷畫成決策菱形
  → 決策菱形只代表人的判斷點；系統自動判斷用虛線箭頭標注條件，不畫菱形

- **使用者描述多角色流程但沒說怎麼分道**
  → 容易把不同角色的步驟混在同一個道裡，交接點不清楚
  → 每個角色一條 swimlane，跨 lane 的箭頭明確代表任務交接；不確定角色分界先問清楚
