# 系統架構圖（System Architecture Diagram）

載入時機：Phase 1 確認視角為「系統架構圖」後讀取。

---

## 定義

描述系統的靜態結構：有哪些系統/服務節點、它們的邊界在哪、彼此如何連接與呼叫。

回答的核心問題：**這個系統由哪些服務組成？邊界在哪？彼此怎麼呼叫？**

---

## 元素白名單

```
✅ 允許
- 系統 / 服務節點（方框，標系統名稱）
- API 介面標示（同步 / 非同步 / REST / event / batch）
- DB 節點（標 DB 名稱；ownership 標示留給 DFD，架構圖不標）
- 外部系統（第三方、GDS、其他 Squad 的服務）
- 部署邊界（虛線框，標 Squad 或 domain）
- 系統間的呼叫箭頭（呼叫方 → 被呼叫方）
- 待確認介面（虛線箭頭 + [待確認] 標注）

❌ 禁止
- 流程步驟框（「建檔」「估算」「上架」等作業動詞序列）
- 決策菱形
- 操作角色（業務、OP、PM、客戶）
- 時間順序箭頭（「先做 A 再做 B」）
- 資料的讀寫權限標示（屬於 DFD）
```

---

## 箭頭語意

- 實線箭頭：已確認的系統呼叫或資料拋轉
- 虛線箭頭：待確認的介面方向或強度
- 箭頭方向：**呼叫發起方 → 被呼叫方**，不是資料流向、不是步驟順序
- 並列模組之間若無直接呼叫關係，**不畫箭頭**

---

## 分層建議（由上至下）

1. 前台 / 使用者介面層（標明渲染責任歸屬）
2. 後台作業系統層（各獨立入口 / 畫面，確認是否有獨立系統）
3. 核心服務 / 模組層（業務邏輯服務、輔助模組）
4. DB 層（標 DB 名稱與 ownership Squad）
5. 跨 Squad 介面層（上游依賴 depends_on、下游輸出 provides_to、待確認）

---

## 常見錯誤（含輸入信號辨識）

每條格式：**[使用者說了什麼]** → [錯誤傾向] → [正確做法]

- **使用者用動詞序列描述流程**（「先建團、再估算、再上架」「流程是 A→B→C」）
  → 容易把作業步驟順序畫成系統箭頭
  → 作業順序屬於作業流圖，架構圖只畫系統節點，不畫步驟序列

- **使用者說「同一個 Squad 的系統都畫進來」或「這幾個系統都屬於 S4」**
  → 容易在並列後台系統之間畫箭頭
  → 並列不等於有呼叫關係，沒有直接 API 依賴就不畫箭頭

- **使用者說「這邊資料流到那邊」「資料從 A 傳到 B」**
  → 容易把資料流向混入系統呼叫箭頭
  → 資料流向歸 DFD；架構圖的箭頭只表示「哪個系統呼叫哪個系統」

- **使用者說「S4 應該會用到 S13 吧」「這兩個系統大概有關係」（推測語氣）**
  → 容易把推測介面畫成實線
  → 方向或強度未確認一律虛線 + [待確認]，不因「應該」就升為實線

- **使用者在節點 subtitle 加「弱依賴」「強相關」「上游」等強度描述**
  → 容易把依賴強度標注留在圖內
  → 依賴強度屬於系統邊界摘要文字，不放進節點框；節點只標系統名稱
