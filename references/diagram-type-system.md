# 系統架構圖（System Architecture Diagram）

載入時機：Phase 1 確認視角為「系統架構圖」後讀取。

---

## 定義

描述系統的靜態結構：有哪些系統/服務節點、它們的邊界在哪、彼此如何連接與呼叫。

回答的核心問題：**這個系統由哪些服務組成？邊界在哪？彼此怎麼呼叫？**

---

## 元素白名單

```
✅ 允許
- 系統 / 服務節點（方框，標系統名稱）
- API 介面標示（同步 / 非同步 / REST / event / batch）
- DB 節點（標 DB 名稱與 owner Squad）
- 外部系統（第三方、GDS、其他 Squad 的服務）
- 部署邊界（虛線框，標 Squad 或 domain）
- 系統間的呼叫箭頭（呼叫方 → 被呼叫方）
- 待確認介面（虛線箭頭 + [待確認] 標注）

❌ 禁止
- 流程步驟框（「建檔」「估算」「上架」等作業動詞序列）
- 決策菱形
- 操作角色（業務、OP、PM、客戶）
- 時間順序箭頭（「先做 A 再做 B」）
- 資料的讀寫權限標示（屬於 DFD）
```

---

## 箭頭語意

- 實線箭頭：已確認的系統呼叫或資料拋轉
- 虛線箭頭：待確認的介面方向或強度
- 箭頭方向：**呼叫發起方 → 被呼叫方**，不是資料流向、不是步驟順序
- 並列模組之間若無直接呼叫關係，**不畫箭頭**

---

## 分層建議（由上至下）

1. 前台 / 使用者介面層（標明渲染責任歸屬）
2. 後台作業系統層（各獨立入口 / 畫面，確認是否有獨立系統）
3. 核心服務 / 模組層（業務邏輯服務、輔助模組）
4. DB 層（標 DB 名稱與 ownership Squad）
5. 跨 Squad 介面層（上游依賴 depends_on、下游輸出 provides_to、待確認）

---

## 常見錯誤

- 把作業步驟順序（建團 → 估算 → 上架）畫成系統箭頭 → 這是作業流，不是系統依賴
- 並列的獨立後台系統之間畫箭頭 → 只有有呼叫關係才畫
- 把「資料往哪流」和「誰呼叫誰」混用 → 架構圖只畫呼叫關係，資料方向歸 DFD
- 強/弱/待確認介面全用實線 → 已確認實線，弱相關細實線，待確認虛線
